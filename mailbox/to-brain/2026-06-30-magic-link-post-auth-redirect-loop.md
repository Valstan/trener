# Грабля: post-auth redirect на публичный лендинг = «петля входа»

## Симптом
Пользователь жмёт magic-link, успешно входит (cookie ставится), но «войти не может —
по кругу гоняет». Объективно вход **срабатывал**: `complete-login` редиректил на `/`,
а `/` — **статический публичный лендинг** с кнопкой «Войти». Залогиненный видел тот же
лендинг → жал «Войти» → `/login` → (в dev почты нет) → тупик. Воспринимается как баг
авторизации, хотя сессия валидна.

## Корень
Две независимые дыры, обе про «куда вести вошедшего»:
1. **Post-auth redirect жёстко на `/`** вместо экрана по роли.
2. **Публичный лендинг `/` не проверяет сессию** — отдаёт «гостевой» вид и залогиненному.
   (Часто специально статичен «чтобы сборка не требовала БД» — и эта оптимизация
   маскирует проблему.)

## Фикс (дёшево)
- Единый хелпер `роль → домашний экран` (`admin→/admin`, `coach→/coach/schedule`,
  `parent→/parent`, гость→`/`).
- `complete-login` ведёт вошедшего по роли (invite-ветка — на онбординг).
- Лендинг `/` → `force-dynamic` + проверка сессии: залогиненного `redirect()` на его
  экран, гостю — прежний лендинг. (force-dynamic сборку НЕ ломает: dynamic-страницы на
  build не исполняются, Postgres при сборке по-прежнему не нужен — снимает исходный
  довод «оставим статичным».)

## Почему шлю
Переносимо на любой role-based Payload+Next с публичным лендингом (трое из нас на этом
стеке 1:1). Особенно если кто-то возьмёт наш magic-link-паттерн (reference R12): мост
сессии один, а «куда вести после входа» — отдельная ловушка, легко проскакивает мимо
ревью (типы зелёные, юнит-тесты на сам логин зелёные, ломается только живой проход).
