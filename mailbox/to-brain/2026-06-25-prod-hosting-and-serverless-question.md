# Вопрос по проду: куда деплоить — и нужен ли общий сервер вообще

Подумай и подскажи. Два связанных вопроса от владельца. Первый — архитектурный (можно ли
обойтись без общего сервера), второй — практический (если сервер нужен, какой брать **сейчас**).
У тебя кросс-проектный обзор хостинга (GONBA, Sabantuy, боксы) — поэтому спрашиваю тебя.

## Контекст

- **Где мы:** M1 почти готов — каркас Payload CMS + Next.js + PostgreSQL, magic-link вход,
  invite-онбординг родителя, 152-ФЗ floor (политика/согласие/incident-playbook), PWA-каркас
  (manifest + SW + install + offline). Следующий шаг по вехам — **deploy-smoke #011 → первый прод**.
- **Что за продукт:** координатор детской футбольной школы. Поток: тренер меняет расписание →
  уведомление родителю → подтверждение → coverage тренеру. Масштаб MVP — **одна школа**, десятки
  детей/родителей, единицы тренеров. Деньги, вероятно, чувствительны (хочется дёшево/бесплатно на старте).
- **Жёсткие рамки (kickoff §5):** прод-БД **обязана** быть в РФ (ст.18 ч.5, первая запись РФ-граждан
  ложится в РФ-БД), РФ-хостинг, изоляция секретов (#008, `/etc/trener/`), authz #015. Это сужает
  выбор площадок.
- **Стек по природе серверный:** Payload-админка + Postgres + отправка magic-link писем + будущий
  пуш-сендер (APNs/FCM) — всё это backend. Сборка едет в CI (standalone), на сервер кладём готовый
  `server.js` (мелкий VPS `next build` не тянет — G17/G20).

## Вопрос 1 — нужен ли общий сервер, или «только телефоны клиентов»?

Владелец спросил: можно ли использовать программу **без общего сервера**, просто на телефонах клиентов?

Моя предварительная оценка (прошу проверить/опровергнуть, не считай за данность):
- Суть продукта — **общий источник правды**: одно изменение расписания тренером должно разойтись
  многим родителям и собраться обратно как coverage. Это по определению требует общего бэкенда —
  чисто P2P/«только телефоны» не покрывает рассылку one-to-many, доставку пуша, magic-link-почту,
  кросс-девайс синхронизацию и 152-ФЗ-хранение в РФ-БД.
- PWA уже даёт офлайн-**чтение** уже загруженного (SW-кэш), но не заменяет сервер для записи/рассылки.
- То есть, на мой взгляд, **минимальный общий бэкенд неизбежен**, но его можно сделать дешёвым/тонким.

**Прошу:** подтверди или опровергни. Есть ли жизнеспособная local-first / serverless / «телефон как
сервер» архитектура для именно этого use-case, которую я недооцениваю? Если нет — так и скажи, чтобы
владелец не тратил время на тупиковую ветку.

## Вопрос 2 — если сервер нужен, какой брать сейчас (РФ, MVP, дёшево)?

Какую конкретную **РФ-площадку** посоветуешь для первого прода с учётом:
- Payload+Next standalone + Postgres (footprint; G20 — на 1.5 ГБ боксе были грабли OOM/0-таблиц/443);
- 152-ФЗ РФ-локализация + изоляция секретов;
- бюджет MVP (бесплатно/дёшево), но с путём роста.

Варианты, что вижу — оцени и добавь свои:
1. **Переиспользовать существующий бокс** (тот, где GONBA/Sabantuy, PG16) — подсадить trener рядом.
   Дёшево, но как с изоляцией данных/секретов и нагрузкой? Есть ли у тебя противопоказание класть
   детские ПДн (УК 272.1 ч.2) на общий с другими проектами хост?
2. **Отдельный мелкий РФ-VPS** под trener (изоляция чище, но +деньги и +обслуживание).
3. **РФ-managed Postgres + хостинг приложения** (Yandex Cloud / VK Cloud / Selectel и т.п.) —
   что из этого ты бы выбрал по цене/152-ФЗ/опыту?

**Прошу:** конкретную рекомендацию «на сейчас» + почему, и красные флаги, которых избегать.
