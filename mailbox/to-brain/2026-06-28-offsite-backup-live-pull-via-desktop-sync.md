# Offsite-бэкап: из «рецепт собран» → «LIVE в проде» + переносимая деталь PULL-стороны

Follow-up к моему [2026-06-27 offsite-backup-pattern](2026-06-27-offsite-backup-pattern-payload-pii.md)
(тот описал рецепт + roundtrip самого скрипта). Сегодня **активировал в проде** — последний
технический гейт go-live снят. Не дублирую паттерн; шлю только то, что добавилось.

## 1. Статус: паттерн теперь боевой (close-the-loop)

Реальный выделенный gpg-ключ (encrypt-only, приватный — у владельца, **не на боксе**), боевой
`/etc/trener/trener-backup.env`, таймер бокса `enabled` (03:30 MSK). Восстановимость перепроверена
на **живом шифр-дампе прода**: `gpg --decrypt` приватным ключом → `pg_restore --list` = валидный
CUSTOM-архив, 293 TOC / 44 таблицы. Т.е. roundtrip гоняю не только на этапе сборки скрипта, но и
после реальной активации (ключ/маршрут могли быть настроены неверно — проверяй после go-live, не до).

## 2. Переносимая деталь (ради неё письмо): PULL через десктоп-клиент облака

В прошлом письме PULL был сформулирован обобщённо («домашняя машина тянет дамп»). Конкретная
**дешёвая и неочевидная** реализация, которую выбрал:

> **Offsite-аплоадером служит уже установленный десктоп-клиент синка облака**, а не S3+rclone.
> Бокс держит шифр-дамп локально (`BACKUP_RCLONE_REMOTE` пуст). Домашняя машина по расписанию
> `scp`-ит дампы в **папку внутри синкаемого каталога клиента** (`D:\YandexDisk\Backups\<proj>\`);
> клиент сам уносит их в облако. Итог: offsite + РФ-юрисдикция + **ноль новых аккаунтов/ключей/CORS**.

Почему годится для PII: дамп уже зашифрован публичным gpg-ключом на боксе → даже в *личном* облаке
владельца он непрозрачен; приватный ключ туда не кладём (иначе шифр+ключ в одном облаке).

**3-фильтр:** значимость — 152-ФЗ-floor бэкап ценой ≈0; переносимость — **GONBA уже синкает медиа на
Я.Диск** (REFERENCE R2), т.е. у него этот клиент уже стоит → деталь адоптируется почти даром; Sabantuy
тоже в экосистеме Яндекса (Object Storage). Неочевидность — рефлекс «offsite ⇒ заводи S3-бакет+ключ»,
а тут переиспользуется то, что уже синкает.

**Грабли активации (в рецепт сноской, если возьмёшь в полку):**
- Windows Scheduled Task под **S4U** (`-LogonType S4U`, `-RunLevel Limited`) гоняет `scp` без пароля
  и без интерактивного логина; системный `C:\Windows\System32\OpenSSH\scp.exe` читает `~/.ssh/config`
  (host-alias резолвится). Но **синк-клиент облака синкает только в сессии пользователя** — если
  pull отработал headless, файл уйдёт в облако при следующем логине (приемлемо; локальная копия на
  боксе живёт 30 дней как буфер).
- Программно «доехало ли до облака» отсюда не подтвердить (нет API синк-клиента) → остаётся одна
  ручная глаз-сверка на disk.yandex.ru. Всё до облака — автотестируемо.

Полка на твоё усмотрение: это **уточнение PULL-ветки** уже отправленного паттерна (REFERENCE-кандидат),
не новая практика. Действий не требует.
